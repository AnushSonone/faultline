This document is the implementation contract for Faultline. Give it to coding
agents before they create files, choose libraries, or change architecture.

Every agent should:

1. Read Sections 1 through 12 before writing code.
2. Read the section for its assigned subsystem.
3. Preserve the canonical schemas and API contracts unless an Architecture
   Decision Record (ADR) explicitly changes them.
4. Implement the smallest complete vertical slice before adding breadth.
5. Add tests, documentation, and measurable acceptance criteria with each
   feature.
6. Never invent benchmark numbers, accuracy numbers, dataset properties, or
   production-scale claims.
7. Never hard-code mock data after the real replay pipeline is available.
8. Keep the core product independent from KillMyCluster.
9. Treat KillMyCluster only as an optional future telemetry source.
10. Prefer a complete, explainable system over a broad but unfinished clone of
    Datadog, Spark, Flink, Jaeger, or Grafana.

The intended final output is not merely a dashboard. It is a working streaming
systems project with:

- Multi-source telemetry ingestion
- Event-time processing
- Out-of-order and late-event handling
- Stateful windowed computation
- Limited streaming SQL planning and execution
- Columnar/vectorized batches
- Incremental service-graph construction
- Evidence-based root-cause ranking
- Checkpoint and recovery behavior
- Synchronized, interactive visual investigation
- Reproducible evaluation on labeled incidents
- A polished public demo and technical write-up

===============================================================================
1. EXECUTIVE SUMMARY
===============================================================================

Faultline is an interactive visual debugger for distributed cloud systems.

It ingests metrics, logs, distributed traces, and deployment/configuration
events. It converts them into a synchronized model of system behavior and lets
a user replay an incident as it unfolded.

The user should be able to:

- Watch requests propagate through a service topology
- Scrub backward and forward through incident time
- See which services became anomalous first
- Compare healthy and failed distributed traces
- Inspect a ranked list of likely root causes
- Click each root-cause candidate to see supporting and contradicting evidence
- Observe conclusions update when late telemetry arrives
- Crash and restart Faultline without losing the active investigation
- Inspect the logical and physical plan of the streaming queries powering the
  visualizations

The project must be useful even if KillMyCluster does not exist. Its primary
datasets and demo environment are independent microservice systems.

KillMyCluster may be mentioned in interviews only as inspiration:

"Building a failure-oriented distributed system made me interested in how
production engineers understand failures across many services. I then built
Faultline as an independent incident-replay and root-cause visualization
engine using public microservice incident datasets and OpenTelemetry."

2. PRODUCT IDENTITY
===============================================================================

Project name:
Faultline

Subtitle:
Interactive Incident Replay and Root-Cause Visualization

One-sentence pitch:
Faultline turns metrics, logs, traces, and deployment events into a synchronized
visual replay of a cloud incident, then ranks likely root causes and exposes the
evidence behind each inference.

Recruiter-friendly pitch:
A visual cloud-debugging platform that replays failures across microservices and
shows how latency and errors propagate through a system.

Engineer-friendly pitch:
A Rust streaming runtime with event-time windows, stateful joins, columnar
execution, checkpoint recovery, and graph-based root-cause ranking, exposed
through a synchronized React visualization interface.

Primary user:
A software engineer, platform engineer, or site reliability engineer trying to
understand a distributed-system incident.

Primary user problem:
Raw logs and dashboards force humans to mentally reconstruct timing, topology,
causality, and request propagation. Faultline creates one synchronized visual
model instead.

Core differentiators:

1. Time synchronization across every view
2. Visual incident replay, not static dashboards
3. Evidence-ranked root-cause inference, not an opaque AI answer
4. Streaming correctness under delayed and out-of-order telemetry
5. Query-engine internals exposed through a visual execution-plan inspector
6. Reproducible evaluation using labeled incident datasets

4. PRODUCT PRINCIPLES
===============================================================================

P1. VISUALIZATION IS THE PRODUCT
The UI is not a decorative dashboard added after the backend. The core product
is a synchronized visual explanation of system behavior.

P2. EVERY INFERENCE MUST BE AUDITABLE
A root-cause score must link to concrete supporting evidence: metrics, spans,
logs, topology, temporal order, or deployment events.

P3. EVENT TIME IS FIRST-CLASS
The system must distinguish when an event occurred from when Faultline
received it.

P4. LATE DATA MUST CHANGE THE STORY
When delayed telemetry arrives, Faultline should either revise the active
incident model or visibly explain why the event was too late to affect it.

P5. COMPLETENESS OVER BUZZWORDS
A small, correct streaming runtime is more valuable than an unfinished clone of
Spark or Flink.

P6. DETERMINISM
Given the same dataset, replay seed, query definitions, and configuration, the
backend must produce the same ordered projections and inference results.

P7. NO BLACK-BOX LLM ROOT CAUSE
Do not use an LLM as the primary inference mechanism. An LLM may summarize
already-computed evidence only after the deterministic core works.

P8. REAL DATA BEFORE SCALE CLAIMS
All displayed incidents must come from a documented public dataset or a
reproducibly generated fault scenario.

P9. ONE CLICK, ALL VIEWS
Selecting a service, trace, time range, deployment, or root-cause candidate must
update every relevant visualization.

P10. NO UNSUPPORTED RESUME METRICS
Every number placed on the resume must be produced by a committed benchmark or
evaluation script and documented in RESULTS.md.

5. PRIMARY USE CASE AND DEMO INCIDENT
===============================================================================

Use case:
An engineer investigates why checkout latency and errors increased after a
software deployment in a microservice application.

Recommended demo environment:
The OpenTelemetry Demo or a replayed RCAEval incident.

Canonical demo incident:
A new recommendation-service version is deployed. A cache-related defect causes
memory growth and garbage-collection pauses. Recommendation spans become slower.
Checkout requests that depend on recommendation become slower. Frontend errors
increase. Some telemetry arrives late. Faultline reconstructs and explains the
propagation.

Demo sequence:

1. The service graph is healthy.
2. A deployment marker appears on the timeline.
3. Recommendation-service memory begins rising.
4. Recommendation-service anomaly cells appear first in the heatmap.
5. Trace particles on affected request paths slow down.
6. Checkout p99 latency crosses an SLO.
7. Root-cause ranking updates:
   - recommendation-service: high confidence
   - checkout-service: lower confidence
   - database: lower confidence
8. The user clicks recommendation-service.
9. Every view filters to supporting evidence.
10. A delayed log event arrives and increases confidence.
11. The causal evidence graph adds:
    deployment -> memory growth -> GC pause -> span latency -> checkout SLO
12. The user compares a healthy and failed trace.
13. The backend is intentionally terminated.
14. The backend restarts from a checkpoint.
15. The active incident, timeline position, stateful windows, and emitted alert
    identities are restored.
16. No duplicate incident alert is emitted.
17. The user opens the query-plan inspector and sees the operators powering the
    p99 latency and anomaly heatmap.

This scenario is the north star. A feature that does not improve this demo or
the underlying technical evidence should be considered lower priority.

8. TECHNICAL STACK
===============================================================================

BACKEND LANGUAGE:
Rust

Why:
- Strong systems signal
- Memory safety
- Async runtime ecosystem
- Good Arrow support
- Suitable for streaming operators, state machines, and performance work
- Adds breadth beyond the owner's existing Go and Python projects

FRONTEND:
TypeScript + React

BUILD TOOL:
Vite or an equivalent lightweight React build setup

GRAPH VISUALIZATION:
Cytoscape.js for the initial service topology and evidence graph

CUSTOM VISUALIZATION:
D3 primitives for scales, axes, brushing, heatmaps, timelines, and trace
waterfalls

BACKEND HTTP/WEBSOCKET:
Axum

ASYNC RUNTIME:
Tokio

SERIALIZATION:
Serde

COLUMNAR DATA:
Apache Arrow RecordBatch through arrow-rs

ARCHIVAL FORMAT:
Apache Parquet

SQL PARSING:
sqlparser-rs

IMPORTANT:
Use sqlparser-rs for syntax parsing only. Do not use DataFusion as the complete
query executor. Faultline must own its logical plan, physical plan, streaming
operators, window state, watermark semantics, and metrics. DataFusion may be
used as architectural reference and as a target for an external contribution.

LOCAL METADATA:
SQLite is acceptable for incident catalogs, user annotations, query metadata,
and benchmark indexes.

STREAMING STATE:
Start with in-memory operator state plus versioned checkpoint files. Use atomic
file replacement and checksums. RocksDB may be introduced later only if state
size or random access justifies it.

OFFLINE DATA PREPARATION AND EVALUATION:
Python + PyArrow + pandas/polars + scikit-learn as needed

FRONTEND TESTING:
Vitest and Playwright

BACKEND TESTING:
Rust unit/integration tests plus property-based tests

CONTAINERIZATION:
Docker Compose for the final local demo

DO NOT ADD BY DEFAULT:
Kafka, Kubernetes, Terraform, Redis, Postgres, Elasticsearch, or a cloud vendor.
Adding infrastructure merely to list it on the resume will dilute the project.

11. DATASET STRATEGY
===============================================================================

PRIMARY BENCHMARK:
RCAEval

Use RCAEval because it provides labeled failure cases across microservice
systems and includes multi-source telemetry. Pin a specific dataset release or
commit in datasets/manifests/rcaeval.yaml.

Do not begin by supporting every RCAEval dataset. Begin with one system and a
small set of incidents representing at least three fault types.

Recommended progression:

Phase A:
- One healthy period
- One memory-related incident
- One CPU/resource incident
- One network/delay incident

Phase B:
- Twenty or more incidents from one system
- Train/validation/test split by incident

Phase C:
- Multiple systems
- Cross-system generalization evaluation

LIVE/GENERATED DATA:
OpenTelemetry Demo

Use the OpenTelemetry Demo to create visually compelling incident recordings.
Use feature flags or documented scenarios to induce reproducible faults.

Generated run metadata must record:

- Git commit of the demo
- Container image versions
- Scenario configuration
- Fault start and end times
- Load-generator configuration
- Random seed
- Ground-truth root-cause service
- Ground-truth fault type
- Expected affected services
- Collection duration
- Faultline converter version

OPTIONAL METRICS-ONLY BASELINE:
Amazon PetShop dataset

Use only as an additional baseline or a simple early pipeline test. It is less
suitable than RCAEval for the final multi-source visual experience because its
main strength is labeled performance-issue metrics.

OPTIONAL SCALE DATA:
Google or Alibaba public cluster traces

Use only after the core project works. These traces may not contain clean RCA
labels. Their purpose is ingestion and aggregation benchmarking, not primary
root-cause accuracy.

DATASET RULES:

1. Never commit large raw datasets to Git.
2. Commit manifests, checksums, download scripts, licenses, and small fixtures.
3. Record exact source versions.
4. Validate timestamps, service names, missing values, and label consistency.
5. Split by entire incident, never by individual event.
6. Prevent healthy and incident slices from the same run from leaking across
   training and evaluation sets.
7. Keep the first scoring method train-free when possible.
8. If weights are tuned, tune only on the training split.
9. Publish exact evaluation splits.
10. Include dataset license notes.

12. CANONICAL TELEMETRY MODEL
===============================================================================

All source-specific data must be normalized into a common envelope.

TelemetryEnvelope {
    schema_version: u16,
    event_id: EventId,
    event_time_ns: i64,
    observed_time_ns: i64,
    ingest_time_ns: i64,
    source_id: String,
    dataset_id: String,
    incident_id: Option<String>,
    environment: String,
    service: Option<String>,
    service_instance: Option<String>,
    host: Option<String>,
    region: Option<String>,
    signal: TelemetrySignal,
    attributes: Map<String, Value>,
    payload: TelemetryPayload,
}

TelemetrySignal enum:

- Metric
- Span
- Log
- Deployment
- Configuration
- Alert
- Annotation
- Control

Metric payload:

MetricPoint {
    name: String,
    kind: Gauge | Counter | HistogramSample,
    value: f64,
    unit: Option<String>,
}

Span payload:

SpanEvent {
    trace_id: String,
    span_id: String,
    parent_span_id: Option<String>,
    operation: String,
    start_time_ns: i64,
    end_time_ns: i64,
    duration_ns: i64,
    status: Unset | Ok | Error,
    peer_service: Option<String>,
    span_kind: Internal | Server | Client | Producer | Consumer,
}

Log payload:

LogEvent {
    severity_number: Option<u8>,
    severity_text: Option<String>,
    body: String,
    template_id: Option<String>,
    trace_id: Option<String>,
    span_id: Option<String>,
}

Deployment payload:

ChangeEvent {
    change_id: String,
    change_type: Deployment | Config | FeatureFlag | Rollback,
    version_before: Option<String>,
    version_after: Option<String>,
    actor: Option<String>,
    metadata: Map<String, Value>,
}

Control payload:

ReplayControl {
    kind: IncidentStart | IncidentEnd | WatermarkHint | CheckpointRequest,
    metadata: Map<String, Value>,
}

IDENTITY RULES:

- event_id must be deterministic for replayed datasets.
- event_id should be a hash of stable source fields plus dataset version.
- trace_id and span_id must remain source-accurate when available.
- service names must pass through a dataset-specific canonicalization table.
- unknown service values must not silently become "other".
- schema_version changes require migration code or a documented compatibility
  break.

TIME RULES:

event_time:
When the event happened in the observed system.

observed_time:
When the telemetry collector observed it, if known.

ingest_time:
When faultlined accepted it.

processing_time:
The current backend clock during execution; do not store as source truth.

- expected_downstream_services

14. REPLAY SYSTEM
===============================================================================

The replay service turns static normalized datasets into live event streams.

Required controls:

- Load incident
- Start
- Pause
- Resume
- Seek
- Stop
- Set speed
- Reset
- Inject disorder
- Inject duplicates
- Inject drops
- Inject a backend crash request
- Request checkpoint

Supported replay speeds:

- 0.5x
- 1x
- 2x
- 5x
- 10x
- 50x
- max-throughput

Replay modes:

1. Faithful
   Preserve source event order based on event_time with documented tie-breaking.

2. Observed
   Preserve observed/arrival order if the source provides it.

3. Adversarial
   Add deterministic delays, disorder, duplicates, and bursts.

4. Benchmark
   Ignore wall-clock pacing and emit as fast as allowed.

DisorderConfig {
    seed: u64,
    max_delay_ms: u64,
    late_event_probability: f64,
    duplicate_probability: f64,
    drop_probability: f64,
    burst_size: usize,
    burst_interval_ms: u64,
}

Rules:

- All random behavior must be seeded.
- The replay manifest must record the disorder configuration.
- Seeking must reset or restore engine state consistently.
- A seek cannot silently retain state from a later time.
- Replay control messages are part of the deterministic event log.

===============================================================================
15. STREAMING ENGINE
===============================================================================

The engine is the technical center of the project.

15.1 INGESTION

Responsibilities:

- Validate schema version
- Assign ingest sequence number
- Deduplicate by event_id
- Route by signal and partition key
- Apply bounded buffering
- Track source health
- Expose ingestion metrics

Recommended partition keys:

- Metrics: service + metric name
- Spans: trace_id
- Logs: service
- Changes: service
- Incident projections: incident_id

15.2 REORDER BUFFER

Each input partition maintains:

- Maximum event time observed
- Current watermark
- Buffered events ordered by event time and sequence number
- Late-event counters
- Duplicate counters

Watermark baseline:

watermark = max_event_time_seen - allowed_lateness

Start with fixed allowed lateness configured per source. Later, experiment with
adaptive lateness based on observed delay distributions.

Watermark invariants:

- A partition watermark never moves backward.
- The global watermark is the minimum active partition watermark.
- Idle partitions must not block progress forever.
- Window finalization occurs only when the relevant watermark passes the
  window end.
- Late events inside the grace period may revise previously emitted results.
- Events beyond the final grace period are retained for audit but do not mutate
  finalized state unless a query explicitly allows revision.

15.3 COLUMNAR BATCHING

Convert validated envelopes into signal-specific Arrow RecordBatches.

Batch flush conditions:

- Maximum row count reached
- Maximum byte size reached
- Maximum batch age reached
- Checkpoint barrier
- Replay pause or seek

Benchmark multiple batch sizes. Do not assume larger is always better.

15.4 OPERATOR MODEL

Conceptual trait:

trait Operator {
    fn id(&self) -> OperatorId;
    fn on_batch(&mut self, batch: RecordBatch, ctx: &mut OperatorContext)
        -> Result<Vec<OperatorOutput>>;
    fn on_watermark(&mut self, watermark: EventTime, ctx: &mut OperatorContext)
        -> Result<Vec<OperatorOutput>>;
    fn snapshot(&self) -> Result<OperatorSnapshot>;
    fn restore(&mut self, snapshot: OperatorSnapshot) -> Result<()>;
    fn metrics(&self) -> OperatorMetrics;
}

Initial physical operators:

- SourceExec
- FilterExec
- ProjectionExec
- HashAggregateExec
- WindowAggregateExec
- PercentileExec
- TemporalJoinExec
- TopKExec
- IncidentSinkExec
- ProjectionSinkExec

15.5 WINDOW TYPES

Tumbling window:
Non-overlapping fixed intervals.

Hopping window:
Fixed-size window emitted at a smaller slide interval.

Session windows are a stretch goal and should not block completion.

Window state key:

QueryId + OperatorId + GroupKey + WindowStart + WindowEnd

Required aggregates:

- count
- sum
- min
- max
- average
- error rate
- approximate p50
- approximate p95
- approximate p99

Percentile implementation:
Start with a documented approximate sketch or bounded reservoir. Include error
evaluation. Do not label an estimate as exact.

15.6 STATEFUL TEMPORAL JOIN

Required use case:
Join service spans or metrics with nearby deployment/configuration changes.

Join semantics:

- Key equality on canonical service
- Event-time interval constraint
- Bounded retention
- Watermark-driven cleanup
- Explicit behavior for unmatched rows
- State-size metrics

Example conceptual condition:

span.service = deployment.service
AND deployment.event_time BETWEEN
    span.event_time - 10 minutes
    AND span.event_time + 1 minute

15.7 BACKPRESSURE

Use bounded Tokio channels.

Every stage must expose:

- Queue depth
- Capacity
- Enqueue wait time
- Batch processing time
- Dropped messages, which should be zero in normal operation
- Current watermark lag
- State size

When a downstream queue is full, upstream work should await capacity rather
than unboundedly allocating memory.

The UI should show backpressure in Query Inspector.

15.8 SCHEDULING

Version 1:
Cooperative async scheduling within one process.

Do not build a distributed scheduler.

Prioritize:

- Control events
- Checkpoint barriers
- Watermark propagation
- Data batches
- Projection updates

Prevent high-volume raw events from starving control operations.

18. INFERENCE ENGINE
===============================================================================

The initial inference system should be deterministic, explainable, and
evaluated.

18.1 INCIDENT DETECTION

An incident begins when one or more SLO or anomaly conditions remain above a
threshold for a configurable number of windows.

Possible signals:

- p99 latency
- error rate
- request throughput drop
- saturation metric
- queue depth
- failed trace rate

Use hysteresis to avoid opening and closing incidents repeatedly.

18.2 BASELINE ESTIMATION

Initial baseline:
Rolling median and Median Absolute Deviation (MAD) by service, metric, and
comparable time context.

Robust z-score concept:

z = 0.6745 * (x - median) / MAD

Handle MAD = 0 explicitly.

Alternative baselines to evaluate:

- Exponentially weighted moving statistics
- Seasonal time-bucket baseline
- Healthy-run reference distribution

18.3 ROOT-CAUSE FEATURES

For each service candidate, compute normalized features in [0, 1].

A. anomaly_strength
How unusual are the service's relevant metrics, logs, and spans?

B. temporal_precedence
Did the service become anomalous before downstream failures?

C. failed_trace_coverage
What fraction of failed traces include this service?

D. critical_path_contribution
How much excess critical-path latency is attributable to this service?

E. downstream_impact
How many affected nodes are reachable downstream?

F. topology_consistency
Does the observed propagation follow known dependency edges?

G. change_proximity
Was there a deployment/config/feature-flag change nearby?

H. log_evidence
Did high-severity or novel log templates increase near onset?

I. persistence
Did the candidate remain anomalous rather than spike briefly?

J. contradiction_penalty
Evidence against causality, such as anomaly beginning after downstream impact.

18.4 INITIAL SCORE

score =
  0.20 * anomaly_strength
+ 0.15 * temporal_precedence
+ 0.15 * failed_trace_coverage
+ 0.15 * critical_path_contribution
+ 0.10 * downstream_impact
+ 0.10 * topology_consistency
+ 0.10 * change_proximity
+ 0.05 * log_evidence
- 0.10 * contradiction_penalty

These weights are a starting hypothesis, not a factual optimum.

Rules:

- Store feature values and contribution values separately.
- Tune weights only on the training split.
- Report untuned and tuned results.
- Perform ablation tests.
- Never hide negative evidence.
- Use the same score implementation for backend output and evaluation.

23. CHECKPOINTING AND RECOVERY
===============================================================================

Checkpoint must capture:

- Session metadata
- Replay position and source offsets
- Global watermark
- Per-partition watermark
- Reorder-buffer state required for correctness
- Active operator state
- Window aggregates
- Temporal-join state
- Service graph aggregates
- Active incident state
- Root-cause feature state
- Emitted alert/evidence IDs
- Projection revision counters
- Query definitions and plan versions

Atomicity strategy:

1. Write to temporary checkpoint directory.
2. Flush files.
3. Compute checksums.
4. Write manifest last.
5. Atomically rename temporary directory.
6. Atomically update LATEST pointer.

Recovery:

1. Read LATEST.
2. Validate manifest and checksums.
3. Restore operators.
4. Restore graph and inference state.
5. Restore source offsets.
6. Rebuild projections or load projection snapshot.
7. Resume after the checkpoint boundary.
8. Deduplicate replayed events by event_id.
9. Do not duplicate incident alerts or evidence IDs.

Failure tests:

- Crash during batch processing
- Crash during checkpoint write
- Corrupted latest checkpoint
- Missing operator state
- Replay of events after checkpoint
- Late event immediately before crash
- Query added before checkpoint
- Query removed after checkpoint

Do not claim exactly-once processing unless every output and recovery path
actually supports it. A safer claim is "checkpoint recovery with idempotent
incident projections."

37. RISKS AND MITIGATIONS
===============================================================================

Risk: UI consumes all project time.
Mitigation: Ship topology, timeline, heatmap, and one trace waterfall before
visual polish. Use stable libraries and fixed layouts.

Risk: The streaming engine becomes too broad.
Mitigation: Support only the operators required by built-in incident queries.

Risk: RCA accuracy is weak.
Mitigation: The project remains valuable through explainable baselines,
evaluation, and visual debugging. Do not overclaim. Use error analysis as a
strength.

Risk: Public dataset formats are inconsistent.
Mitigation: Build adapters behind one canonical schema and pin versions.

Risk: Trace data is incomplete.
Mitigation: Represent missing spans explicitly and include completeness as
evidence confidence.

Risk: Checkpointing delays progress.
Mitigation: Implement operator snapshot round trips early, then coordinate
globally.

Risk: The visual graph becomes noisy.
Mitigation: Use stable layout, aggregation, filtering, progressive disclosure,
and service search.

Risk: Project appears like a dashboard clone.
Mitigation: Expose event-time semantics, query plans, late-event revisions,
root-cause features, critical paths, and recovery behavior in the UI and docs.

Risk: Agents create incompatible modules.
Mitigation: Enforce schemas, protocol DTOs, ticket dependencies, and ADRs from
this file.

42. FINAL PROJECT THESIS

Faultline should communicate one coherent thesis:

"Distributed incidents are difficult because evidence is fragmented across
time, services, and telemetry types. I built a streaming engine that reconstructs
that evidence into a synchronized visual model, ranks likely root causes, and
shows exactly why each conclusion was reached, even when events arrive late and
the analysis process itself fails."

That thesis is more important than the number of features.

The best final project is not the one with the most technologies. It is the one
where:

- the problem is obvious,
- the visuals are memorable,
- the inference is defensible,
- the architecture is deep,
- the benchmarks are reproducible,
- the failures are honest,
- and every implementation choice creates interview material.
